# Wiki-AI — Instruções para Desenvolvimento

## Visão Geral

Este projeto implementa a arquitetura de "LLM Wiki" de Karpathy, onde um sistema compila e mantém incrementalmente um repositório persistente de conhecimento em markdown. O objetivo é transformar conhecimento bruto em síntese estruturada, compondo no tempo.

---

## Três Camadas de Conhecimento

### 1. Fontes Brutas (`fontes/`)
- Código-fonte original, documentação, artigos, links
- **Imutável** — não são alteradas pela LLM
- Catalogadas em `fontes/catalogo.md`
- Cada fonte tem ID, tipo, tags, e referência

### 2. Páginas Wiki (`wiki/pages/`)
- Conhecimento compilado e sintetizado
- **Mantidas pela LLM** incrementalmente
- Cada página trata de um conceito, padrão ou decisão arquitetural
- Indexadas em `wiki/index.md`

### 3. Sessões de Trabalho (`sessoes/`)
- Registro incremental de conversas e decisões
- Cada sessão tem resumo que alimenta a wiki
- Catalogadas em `sessoes/indice.md`

---

## Princípios de Design

### Conhecimento Persistente vs. Transiente
- **Nunca** re-derive a mesma conclusão em cada query
- **Compile uma vez**, reutilize sempre
- Quando uma ideia surge, sintetize para a wiki

### Divisão de Trabalho
- **Humano**: cura fontes, faz perguntas, valida decisões
- **LLM**: sumariza, cruza referências, mantém índices, faz linting

### Modulação de Custo
- Leitura de fontes é "caro" (contexto) → compilar para wiki
- Manutenção é "tedioso" → automatizar na LLM
- Síntese é "criativa" → LLM manipula relações

### Versionamento Git
- Toda mudança é um commit
- Wiki funciona como repositório vivo
- Histórico de decisões está em `git log`

---

## Fluxo de Trabalho

### Ao Receber Conhecimento Novo

1. **Catalogar fonte** → `agent-fontes`
   - Se é documentação, código, ou artigo, registre em `fontes/catalogo.md`

2. **Processar sessão** → Converse normalmente, registre em sessão

3. **Compilar para wiki** → `agent-wiki`
   - Resume achados em novas páginas ou atualiza existentes
   - Cruza referências com conhecimento anterior
   - Atualiza índices

4. **Lint da wiki** → `agent-wiki`
   - Procura contradições entre páginas
   - Identifica conceitos órfãos (fontes sem síntese)
   - Sugere consolidações

### Ao Responder Perguntas

1. **Primeiro**, busque em `wiki/pages/`
2. **Se não encontrar**, procure em `fontes/`
3. **Se ainda não souber**, registre a lacuna para compilação futura
4. **Nunca** use conhecimento externo para preencher — pergunte ao usuário

## Boas Práticas para Páginas Wiki

### Estrutura de Página

```markdown
# [Conceito ou Padrão]

## Resumo
[1-2 linhas explicando o essencial]

## Por Quê?
[Contexto, motivação, problema que resolve]

## Como Aplicar
[Passos práticos, exemplos, quando usar]

## Referências Cruzadas
- [[Página relacionada 1]]
- [[Página relacionada 2]]

## Fontes
- Fonte ID: X — [descrição]
```

### Convenções

- **Nomes curtos e descritivos** — "Logging Estruturado", não "Sistema de Logs para Produção"
- **Sem redundância** — cada ideia uma vez; cruzar não duplicar
- **Links internos** — `[[Página]]` para criar rede de conhecimento
- **Citações explícitas** — sempre aponte qual fonte embasa afirmação
- **Exemplos concretos** — princípios abstratos + código/caso real

---

## Checklist de Qualidade de Wiki

Antes de confirmar compilação:

- [ ] Cada página tem claro resumo de 1-2 linhas
- [ ] Explicação do "por quê" está presente
- [ ] Recomendações práticas ("como aplicar") existem
- [ ] Nenhuma página está órfã (sem referência cruzada)
- [ ] Índice foi atualizado com nova página
- [ ] Nenhuma contradição com páginas existentes
- [ ] Todas as fontes usadas estão em `fontes/catalogo.md`

## Interação com Agents

### agent-sessoes
- Abre sessão ao iniciar conversa
- Registra interações dentro de sessão
- Encerra e sumariza antes de wiki update

### agent-wiki
- Compila sessões em páginas
- Atualiza `wiki/index.md`
- Linta para contradições e lacunas

### agent-fontes
- Registra nova fonte em catálogo
- Associa fontes às páginas que as usam
- Mantém histórico de referências

---

## Antes de Cada Commit

1. Confirme que **todas as sessões foram processadas** pela wiki
2. Confirme que **índices estão atualizados**
3. Confirme que **nenhuma contradição existe**
4. Confirme que **humano validou as mudanças**

Commit deve ter mensagem clara: `feat: [o que foi sintetizado] — sessão #X`

---

## Quando Algo Não Se Encaixa

Se um pedido não se encaixa neste pipeline:

1. Pergunte: **"Isso é novo conhecimento para compilar, ou revisão do que já existe?"**
2. Se novo → registre para compilação posterior
3. Se revisão → valide com usuário antes de alterar

O objetivo é **conhecimento estruturado e composto no tempo**, não resposta rápida.
